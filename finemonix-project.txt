\section{Phase 1 --- Data Foundation (Days 1--6)}

\begin{phasebox}{phase1}{phase1bg}{%
  PostgreSQL schema $\cdot$ Tally/GST/bank parsers $\cdot$ Entity resolution $\cdot$ Celery pipeline $\cdot$ Auth%
}
Everything else in NeevFinance is downstream of Phase 1. No ML model, no API endpoint, and
no frontend feature can work until the data ingestion pipeline is solid and the schema is locked.
Do not rush this phase --- a messy schema here means painful refactoring in Phase 3 and 4.
\end{phasebox}

\vspace{0.5em}

  \item Write Python parser: extract all RECEIPT/PAYMENT/JOURNAL vouchers

\begin{promptbox}
I am building NeevFinance. Day 3 -- Tally XML parser.

Tally ERP 9 and TallyPrime export data as XML. I need a Python
parser that:
1. Accepts a Tally XML file (TALLYMESSAGE -> COMPANY -> LEDGER /
   VOUCHER nodes)
2. Extracts all transactions (RECEIPT, PAYMENT, JOURNAL, CONTRA
   voucher types) into my transactions table schema
3. Extracts invoice records (SALESVOUCHER, PURCHASEVOUCHER)
   into my invoices table schema
4. Extracts client/vendor names from LEDGER master records

My transaction schema:
id, business_id, date, amount, direction(in/out), category,
counterparty_id, source='tally', raw_description

Important edge cases to handle:
- Multi-currency entries (convert to INR)
- Cancelled vouchers (ISOPTIONAL = Yes, skip these)
- Group ledger hierarchies (Sundry Debtors = clients,
  Sundry Creditors = vendors)
- Same client appearing with slightly different names

Give me the complete parser class with error handling.
No frontend code.
\end{promptbox}

I am building NeevFinance. Day 4 -- GST JSON and bank CSV parsers.

\section{Phase 2 --- Cash Flow ML (Days 7--12)}

\begin{phasebox}{phase2}{phase2bg}{%
  Feature engineering $\cdot$ LSTM $\cdot$ Prophet baseline $\cdot$ Forecast API $\cdot$ Scenario planner%
}
The cash flow forecasting engine is the heart of NeevFinance. It uses an LSTM for users with
6+ months of data and falls back to Prophet for new users. The scenario planner is what makes
the demo compelling --- a live slider that shows how a client delay affects your cash in real time.
\end{phasebox}

% ----- DAY 9 -----
\begin{daycard}{phase2}{phase2bg}{9}{LSTM forecasting model}

\small\color{mutedtext}PyTorch LSTM for users with 6+ months of transaction history\par\vspace{6pt}

\tagml

\subsection*{Tasks}
\begin{tasklist}
  \item Build PyTorch LSTM model for sequence-to-sequence cash flow forecasting
  \item Write DataLoader with sliding window: 60-day input to 90-day output
  \item Train loop with early stopping (patience=10) and learning rate scheduler
  \item Generate uncertainty bands via MC Dropout (30 forward passes)
  \item Evaluate on test set: MAE, compare to Prophet baseline
  \item Save model checkpoint and write inference function
\end{tasklist}

\begin{promptbox}
I am building NeevFinance. Day 9 -- LSTM model (PyTorch).

Architecture:
- Input: 60 days of features
- Output: 90-day forward prediction of daily cash balance
- Model: 2-layer LSTM (hidden_size=128, dropout=0.2) -> Linear head
- Loss: Huber loss
- Uncertainty: MC Dropout -- 30 forward passes with dropout enabled

Training:
- Sliding window: 60-day inputs, 90-day targets
- Batch size 32, Adam lr=1e-3, ReduceLROnPlateau
- Early stopping patience=10 on validation MAE
- Min 6 months historical data required

Inference output:
forecast_df with: date, predicted_balance, p10_balance, p90_balance

Weekly Celery retraining task:
retrain_lstm(business_id) -> loads latest transactions ->
retrains -> saves checkpoint

I build all visualization (area chart with confidence bands) myself.
\end{promptbox}

\end{daycard}

%  APPENDIX C --- KEY INTERVIEW TALKING POINTS
% ============================================================
\section*{Appendix C --- Key Interview Talking Points}
\addcontentsline{toc}{section}{Appendix C --- Key Interview Talking Points}

\begin{enumerate}[leftmargin=1.5em, itemsep=6pt]

\item \textbf{SHAP additivity for O(1) what-if inference}\\
SHAP values are additive: \texttt{prediction = base\_value + sum(shap\_values)}. When one feature changes, you only need to recompute that feature's SHAP contribution. The pre-computed decomposition is cached in Redis, and the delta is approximated from a single fast inference pass (5ms), not a full model re-run. This is why the what-if simulator responds in under 100ms.

\item \textbf{Why Isolation Forest rather than a supervised anomaly model}\\
There are no ground-truth labels for ``anomalous quarter'' in corporate financial data. Collapses are rare and labelling them requires hindsight. Isolation Forest learns normality from each company's own history (contamination = 0.05) and flags statistical deviations --- this is company-relative, not cross-sectional, which eliminates false positives from genuinely aggressive business models.

\item \textbf{Handling the LSTM cold-start problem}\\
New users have fewer than 6 months of data. The solution is a two-model system: Prophet (a simpler, interpretable model) handles cold-start users and displays wider confidence intervals with an explicit accuracy warning. As the LSTM accumulates 180+ training samples, it automatically takes over. Users see the forecast improve over time, which is itself a product engagement mechanic.

\item \textbf{Entity resolution across noisy Indian financial text}\\
The same promoter appears as ``Rajesh Kumar Sharma'', ``R.K.\ Sharma'', and ``Rajesh Sharma'' across 10 years of PDFs. A two-pass approach is used: exact CIN/PAN match first, then RapidFuzz token-sort ratio for company names, then sentence embedding cosine similarity for person names. The entity\_aliases table is human-correctable via an admin UI for edge cases.

\item \textbf{Why one unified data pipeline for three features}\\
The cash flow data is credit data. The same Tally transaction history that trains the LSTM also computes the loan eligibility features. The same client concentration metric that flags a cash risk also informs the SHAP explanation on the loan page. No redundant data entry and cross-module features (e.g.\ a watchlist alert on a client who is also your highest-concentration revenue source) are only possible because of the shared entity resolution layer.

\end{enumerate}
